このリポジトリを読んでいて、一番面白かったのは理論の中身より先に、運営の設計が見えてきたことだった。

普通、GitHubのリポジトリ管理といえば、コード、Issue、README、たまにCIくらいを想像する。  
でも `kesson-driven-thinking` では、リポジトリ自体がもっと別のものとして扱われている。  
コードや文章の保管庫ではなく、**人間とAIが忘れ、迷い、引き継ぎ、止まり、再開するための運営OS**になっている。

しかも面白いのは、それが大げさなインフラで実現されていないことだ。  
`base/text/meta/system-architecture-public.md` を読むと、物理環境はかなり素朴で、基本は iMac 1台とインターネット接続で回している。  
豪華な基盤があるから成立しているのではない。  
**どの情報を、どこに、どんな粒度で置くか**の設計で成立している。

## READMEが説明書ではなく、起動シーケンスになっている

このrepoでは、ドキュメントが「読めば理解が深まる資料」では終わっていない。  
読む順番まで含めて、実行時の振る舞いが設計されている。

たとえば起点は `docs/README.md` で固定されている。  
そこから `CLAUDE.md`、`.cache/session/state.md`、必要に応じて `operations.md`、`catalog.md`、`quality-management.md`、各ディレクトリの `README.md` や `SKILL.md` に進む。  
つまりここでは README が飾りではなく、**ブートローダー**として機能している。

この設計が効いているのは、AIが全部を覚え続けてくれる前提を捨てているからだ。  
LLMはセッションをまたぐと忘れるし、同じrepoでも読み始める位置によって理解がかなり変わる。  
だから「最初に何を読むか」を管理する。  
これは地味だけど、AI協働ではかなり重要な設計だと思う。

しかも `docs/README.md` には Context Load Manager があり、Tier 0, 1, 2, 3 と段階ロードが定義されている。  
全部読ませるのではなく、**必要になるまで読ませない**。  
ここは最近よく言われるコンテキストエンジニアリングそのものだけれど、このrepoでは概念ではなく運用手順として埋め込まれていた。

## 状態管理を1ファイルに集約しない

もう一つ特徴的なのは、状態管理を「巨大な1枚のダッシュボード」に寄せていないことだ。

普通は state を一元化したくなる。  
でもこのrepoでは、そうしていない。  
代わりに、状態を速度と役割で分けている。

ざっくり言うとこうだ。

| 層 | 主な置き場 | 役割 |
|---|---|---|
| 横断の再開点 | `docs/README.md`, `CLAUDE.md`, `.cache/session/state.md` | どこから再開するか、何を守るか |
| 遅い状態 | 各ディレクトリの `README.md`, `skills/*/SKILL.md` | その場所がなぜ存在するか、今も有効か |
| 速い状態 | `.cache/backlog.md`, `.cache/inbox/`, `.cache/active/`, session log | いま何が動いているか、どこで止まったか |
| 判断の履歴 | `docs/decision-log/` | なぜそのルールや構造になったか |

この分け方がいいのは、変化の速度が違うものを無理に同居させないことだ。  
「このディレクトリは何のためにあるか」はそんなに頻繁には変わらない。  
でも「このセッションでCLIがどこまで進んだか」はすぐ変わる。  
それを同じファイルで持つと、すぐ stale になる。

`docs/decision-log/2026-03/LOG-20260309-002_state-management-deliberation.md` を読むと、このrepoは「どのファイルが正しいか」よりも、**どの項目を誰が持つか**で整合性を設計しようとしている。  
Single Source of Truth を「1ファイル」ではなく、**1項目1オーナー**で考えているのが面白い。

これはソフトウェアの状態管理というより、組織設計に近い。  
実際そのログでも、repoは決定論的な関数群というより、人間組織に近い不確実系として捉えられていた。

## ドキュメントが仕様書ではなく、ハーネスになっている

このrepoでは、管理文書が受動的な説明資料ではない。  
実行時の制約と検証を担う**ハーネス**になっている。

特にそれが見えるのが、以下の4枚だ。

- `docs/README.md`: 入口、起動順、委任の考え方
- `docs/operations.md`: routing、ファイル操作方針、セッション分割、inbox運用
- `docs/catalog.md`: 何がどこにあるか、どういう条件で読むか
- `docs/quality-management.md`: 不変の品質基準とOPSテスト

この4枚は「説明文」ではなく、運用時に読み返される前提で書かれている。  
たとえば `quality-management.md` には OPS テストがあり、管理フローそのものに対しても「これで本当に回るか」を問う。  
つまり成果物だけでなく、**運営そのものがテスト対象**になっている。

ここはかなり珍しいと思う。  
たいていのrepoでは、品質チェックはコードか文章にかかる。  
でもこのrepoでは、READMEの構造、ハンドオフ、状態同期、指示書の完了条件まで検査対象に入っている。  
管理そのものを成果物として扱っている。

## LLM前提のrepoは、DRYだけでは足りない

このrepoで個人的に一番面白かったのは、冗長性に対する考え方だった。

普通の設計では、同じルールを複数箇所に書くと「重複だ、整理しよう」と言いたくなる。  
でもLLMが混ざると、その美学だけではうまくいかない。

`docs/README.md` や `.claude/rules/design-principles.md` にある L7 は、まさにそこを扱っている。  
ルールを一箇所に正規化することより、**必要な入口でちゃんと踏ませること**を重視している。

理由は単純で、LLMは「近くにあるから読む」わけではないからだ。  
ファイルが存在していても、辿り方が明示されていなければ普通に踏み外す。  
だから README を各所に置き、SKILL を入口にし、指示書にも確認項目を書く。  
人間には少しくどく見えても、AI込みの運用ではその冗長性が耐障害性になる。

これはコードのDRY原則に反しているようでいて、実は対象が違う。  
コードの重複削減ではなく、**認知の失敗を防ぐための冗長化**である。

## 「AIに任せる」と「AIに主権を渡す」を分けている

AIを多用するプロジェクトで、いちばん事故になりやすいのはここだと思う。  
便利だからといって、判断権まで曖昧にすると、あとで誰も責任の所在を説明できなくなる。

このrepoはその点がかなり明確だ。  
`CLAUDE.md` には委任レベルがあり、

- AIが自律実行してよいこと
- 提案まではしてよいが承認が必要なこと
- pjdhiro専権で、AIは促すだけのこと

がはっきり分けられている。

しかもこれは抽象的な倫理宣言ではなく、ファイル操作やルール変更、理論採否、公開判断まで具体化されている。  
さらに `docs/operations.md` には LLM Routing Table があり、タスクの種類ごとに DT App、Claude Code CLI、ChatGPT、Gemini の使い分けまで定義されている。

ここで大事なのは、「最強モデルを全部に当てる」発想ではないことだ。  
そうではなく、**どのワークフローに誰を置くか**で考えている。  
理論対話、メタ管理、evidence調査、UI実装、git操作で担当が違う。  
これはモデル比較表より、ずっと実務的なAI運用だと思う。

## backlog, inbox, active, log が時間軸で分業している

`.cache/` 配下の構造もかなりよくできている。

- `.cache/backlog.md` は積み残し全体の俯瞰
- `.cache/inbox/` はこれから投入する指示書のキュー
- `.cache/active/` はいま着手中の TASK
- `.cache/session/state.md` は再開点
- `.cache/session/log-*.md` は何をどう判断したかの時系列ログ

この分け方は、Todo管理を細かくしているというより、**時間の向きごとに置き場を分けている**感じがある。

- backlog は「まだ終わっていない全体」
- inbox は「次に実行する受け渡し」
- active は「今このタスクで使っている作業面」
- log は「終わったあとに残す復元可能な履歴」

AI協働で本当に怖いのは、マージ競合そのものよりも、  
「なぜこの指示書がここにあるのか」  
「このタスクは止まっているのか終わったのか」  
「前回のセッションでどこまで進んだのか」  
が分からなくなることだ。

このrepoはそこをかなり自覚的に潰している。  
主敵をコード衝突ではなく**認知的な再開失敗**として扱っているのが、普通のプロジェクト管理との大きな違いだと思った。

## これは誰に役立つか

このやり方が特に効きそうなのは、次のようなプロジェクトだと思う。

### 1. ソロプレナーや個人研究者

自分ひとりで複数のAIを使っていると、問題は「作業量」より先に「頭の再起動コスト」になる。  
何をどこまで考えたか、どの判断が仮置きか、何を保留にしたかを repo に逃がしておけると、翌日も続けやすい。

### 2. 正解が最初から決まっていない仕事

理論構築、リサーチ、設計、長期の文章制作のように、途中で問いの形そのものが変わる仕事では、  
「done か not done か」だけでは管理できない。  
このrepoのように、保留、再開、委任、専権判断を分けて持つ設計はかなり相性がいい。

### 3. 小規模な人間+AIチーム

人数が少ないチームほど、CIやワークフローエンジンより前に、  
「このフォルダ何のためにあるんだっけ」  
「このAIにはどこまで任せていいんだっけ」  
を失いやすい。  
README、SKILL、decision-log を運用の入口として整えるやり方は、そのまま効くはずだ。

逆に、短期の小さな実装だけを回す repo には少し重い。  
この設計は、曖昧さ、再開コスト、AIの忘却、判断境界の問題が大きいプロジェクト向けだと思う。

## 面白かったのは、repo管理そのものが研究対象になっていること

このrepoでは、プロジェクト管理が裏方ではない。  
`docs/decision-log/` に管理の deliberation が残され、`quality-management.md` には運用整合性テストがあり、`state.md` と `backlog.md` の同期までルール化されている。

つまり「いい理論を作るために管理が必要」なのではなく、  
**AIと何かを作るなら、管理設計そのものが中核成果物になる**という認識がある。

ここはかなり本質的だと思う。

LLM時代のrepo管理は、コードをどう整理するかだけではなく、  
忘却にどう耐えるか、  
判断権をどう守るか、  
未解決をどう保持するか、  
人間とAIの間でどの粒度の文脈を受け渡すか、  
まで含めて設計しないと回らない。

このrepoは、そのことをかなり早い段階で自覚して、ファイルとして実装している。  
だから読んでいて、「きれいな管理」より先に「生き延びるための管理」だと感じた。  
その感じが、すごくよかった。
